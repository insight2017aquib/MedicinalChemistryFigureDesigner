# Design Principles

## Purpose

Define the guiding principles that govern architectural and documentation decisions across the platform.

## Scope

**In scope:**

- Platform design values
- Constraints on content generation
- Modularity and extensibility rules
- Quality standards for documentation

**Out of scope:**

- Visual design tokens (see `styles/`)
- Scientific accuracy criteria
- Tool-specific implementation details

---

## Core Principles

### 1. Architecture First, Content Later

The repository defines structure, contracts, and extension points. Scientific content, domain knowledge, and journal guidelines are supplied by maintainers and users—not generated by the platform.

### 2. No Fabrication

The platform must never invent:

- Biological or chemical facts
- Journal or publisher submission guidelines
- Worked scientific examples

All domain content requires explicit user or maintainer input with attribution.

### 3. Modular by Default

Every capability lives in a discrete module with a clear boundary:

- One README per module explaining purpose and scope
- Cross-references instead of duplication
- Additive changes preferred over restructuring

### 4. Backward Compatibility

New platform layers extend existing modules without breaking their contracts. The v0.1 scaffold directories remain valid and functional across all v0.x milestones.

### 5. Specification Over Implementation

Prefer documented schemas, templates, and checklists over executable code until the milestone requires implementation (v0.4+ for FSL, v0.7+ for validation engine).

### 6. Validate Before Export

No figure specification or asset proceeds to export without passing the validation module. Validation checks structure and compliance—not scientific correctness.

### 7. Traceable Provenance

Every figure specification, knowledge pack entry, and exported asset should support metadata tracking: source, version, template, and style references.

### 8. Accessible by Design

Accessibility rules are mandatory constraints, not optional enhancements. See `rules/accessibility.md`.

---

## Documentation Standards

- Every file includes Purpose and Scope sections
- Placeholder content is explicitly marked as TODO or placeholder
- Cross-references use relative paths within the repository
- Mermaid diagrams for architecture; tables for structured data

---

## Extension Guidelines

When adding a new module or component:

1. Create a README with Purpose, Scope, and TODO sections
2. Register the module in root `README.md` and `docs/Architecture.md`
3. Add a milestone entry in `docs/DevelopmentRoadmap.md` if applicable
4. Do not embed scientific content in platform documentation
5. Update `docs/Changelog.md` with the change
